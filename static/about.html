About this project

A notebook doesn't prove you can ship.

This app takes a brain MRI, classifies it as glioma, meningioma, pituitary, or no tumor, and shows a Grad-CAM heatmap of what the model actually looked at. Here's why it exists, how the pipeline works, and the mid-build platform switch that nearly derailed it.

01Origin

Why this exists

The model started as a Jupyter notebook: a small CNN trained on brain MRI scans to tell four classes apart, run cell-by-cell on a personal machine until it hit a respectable test accuracy. That's a fine way to learn, but it doesn't demonstrate much beyond "I can write Python in a notebook." Nobody can open a link and try it. There's no API, no deployment, no failure handling — none of the things that separate a modeling exercise from a shipped application.

This project is that notebook turned into something a recruiter, another developer, or anyone with a browser can actually use: a FastAPI backend serving real-time inference, a Grad-CAM layer so the prediction isn't a black box, and a deployment pipeline where pushing to main is the only way the live app changes.

There's a twist worth admitting: the original trained model file was lost. The save path and the environment that produced it were both gone by the time this build started, so step zero wasn't "wire up the API" — it was retraining the notebook from scratch to get a usable model artifact back.

02The model

How it works

The network is deliberately simple: three Conv2D + MaxPooling blocks (64 / 64 / 128 filters), flattened into a Dense(128, ReLU) layer with dropout, ending in a 4-way softmax. Trained on the Kaggle brain-tumor-mri-dataset at 256×256 resolution. Nothing exotic — the point of this build was never a state-of-the-art architecture, it was proving the surrounding engineering.

  1. Decode & resizeThe uploaded image is decoded and resized using TensorFlow's own ops — not PIL. The model was trained on TF's internal resize behavior, and a PIL reimplementation uses different interpolation, which quietly skews accuracy between training and serving.
  2. NormalizePixels are rescaled to a 0–1 float range, matching training preprocessing exactly.
  3. PredictThe CNN, loaded once at process startup, returns a class label plus a confidence score for all four classes — not just the top guess.
  4. ExplainGrad-CAM finds the model's last convolutional layer programmatically (never a hardcoded layer name), traces gradients back through it, and overlays a heatmap so you can see what drove the call.

On honesty: the training notebook doesn't fix a random seed for weight initialization (only the train/validation split is seeded), so accuracy varies a little between runs. The deployed model's numbers, from the run actually hosting the live app:

That test figure is below the original notebook's 93.7%, and it's reported here rather than quietly swapped for the better-sounding one. Re-running training with a fixed seed to close that gap is tracked as a follow-up, not smoothed over.

03Under the hood

Architecture

The backend is split into small, independently testable modules rather than one monolithic route handler: preprocessing (bytes in, normalized array out), inference (array in, prediction out), and Grad-CAM (array in, heatmap out). The FastAPI route layer just wires them together and translates failures into clean HTTP errors — a bad upload returns a 400, never an unhandled crash.

The trained .keras file itself never touches git. It's hosted on a public Hugging Face Hub model repo and pulled into the Docker image at build time, pinned to a specific commit revision — so an unrelated code change can never silently swap the live model, and the running container has zero runtime dependency on Hub availability.

04The detour

A deployment plan that changed mid-build

The original plan was Hugging Face Spaces on its free Docker SDK tier — a natural fit, since the model was already going to live on the HF Hub. Mid-build, Hugging Face changed policy to require a PRO subscription to create a Docker SDK Space on a personal account, even on free CPU hardware. Paying a subscription fee for a portfolio demo didn't make sense, so the deploy target moved.

Google Cloud Run ended up a better fit anyway: it runs the exact same Dockerfile unmodified, its free tier's memory ceiling is well above what TensorFlow needs (this app runs at 2 GiB / 2 vCPU), and it scales to zero when nobody's using it. The only cost is a GCP billing account on file — no recurring charge at this app's traffic level.

GitHub stays the single source of truth throughout: a GitHub Actions workflow builds the image, pushes it to Artifact Registry, and deploys a new Cloud Run revision on every push to main. Nothing gets deployed by hand.
